Although I find the constant red underlines an eyesore from ESLint errors while developing, having proper coding standard
has helped me significantly as it allows me to write uniform code without needing to think.  By pairing ESLint with the
introduction of using an integrated development environment (IDE) such as IntelliJ IDEA, it has had a remarkable impact
on my productivity by offering great tools and features.  The use of ESLint has definitely forced me to write code in a 
standardized manner without having to constantly think about what I should be doing, rather I can code first and then think 
about correction.

I will admit, the process of setting up and configuring IntelliJ IDEA so that it could start up in less than 10 seconds was 
definitely a pain.  The constant adjustment of settings and getting familiar with the numerous features, options, and 
general usage of the IDE took a bit time.  With the first couple of programs we were writing in the IDE, I was more focused on
writing code that was compliant with ESLint rules just to get that green checkmark.  However, the constant reminder of the 
ESLint rules with its red underlining has forced me to develop a habit of writing in a standard code format.

Despite the initial fustrations working with IntelliJ IDEA and ESLint, the use of maintaing coding standards allows for overall
readability and therefore allows for maintainability and higher code quality.  This doesn't mean much for when we are working
on programs that we build for ourselves, however a standard becomes especially important when we work on projects with another 
developer and even more so with a large scale project with hundreds of developers.  Therefore it is critical that we maintain
a consistent coding standard and since a tool like ESLint does all the hard work of finding the errors, why wouldn't you use it?
